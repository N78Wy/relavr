# Quest 3 推流发送端需求文档（简版）

## 1. 目标

开发一个运行于 **Meta Quest 3** 的发送端应用，完成头显画面与音频采集、编码与实时推流。
技术路线固定为：**Android App / OpenXR App → MediaProjection 抓画面 → AudioPlaybackCapture 抓音频 → MediaCodec 编码 → WebRTC 推流**。Meta Horizon OS 支持 Android 应用形态，Quest 也支持 OpenXR；Quest 上已开放 MediaProjection 用于 casting / live streaming / screen sharing。([Meta for Developers][1])

## 2. 当前范围

本阶段**仅实现发送端**，不实现浏览器端或 Android 接收端业务功能。
发送端需输出标准 WebRTC 音视频流，供后续浏览器 / Android 接收端接入。WebRTC 本身支持实时音视频与主流平台互通。([WebRTC][2])

## 3. 兼容性要求

* 发送端运行平台：**Meta Quest 3**。Quest 3 属于 Meta 当前支持的 Android 设备开发范围。([Meta for Developers][3])
* 最终接收端兼容目标：**Android 5.0+**。为适配该目标，发送端的默认编码能力必须优先覆盖 Android 5.0+ 上普遍可解码的主流格式。Android 的 MediaProjection 也自 Android 5.0 起提供。([Android Developers][4])

## 4. 功能需求

### 4.1 画面采集

* 发送端必须基于 **MediaProjection** 采集 Quest 3 显示内容。
* 采集流程必须符合系统授权要求，不允许绕过用户同意。
* 画面采集输出需可直接接入编码器输入 `Surface`。([Meta for Developers][5])

### 4.2 音频采集

* 发送端必须支持通过 **AudioPlaybackCapture** 采集播放音频。
* 音频采集应作为独立模块，便于后续做开关控制、静音策略与异常降级。([Android Developers][6])

### 4.3 编码

* 发送端必须基于 **MediaCodec** 实现硬件优先编码。
* 发送端必须支持**多种主流编码格式**，至少包含：

  * **H.264 / AVC**（必选，默认主格式）
  * **H.265 / HEVC**（可选）
  * **VP8 / VP9**（可选，按设备能力启用）
* 为兼容未来 **Android 5.0+** 接收端，默认推流配置必须优先使用 **H.264**；其余编码格式作为能力扩展与协商项。Android 官方支持格式文档列出了 AVC、HEVC、VP8、VP9 等平台支持情况。([Android Developers][7])

### 4.4 推流

* 发送端必须通过 **WebRTC** 输出实时音视频流。
* 推流模块需与采集、编码模块解耦，便于后续增加接收端、信令服务、重连与带宽自适应。
* 本阶段只要求完成单路实时推流，不要求实现多房间、鉴权、录制、转码。([WebRTC][2])

## 5. 非功能要求

* 默认实现应以**低延迟实时预览**为目标。WebRTC 适用于实时视频、音频与数据传输。([WebRTC][2])
* 编码与推流链路需支持后续扩展码率、分辨率、帧率和编码格式切换。`MediaFormat` / `MediaCodec` 支持通过参数化方式配置媒体格式。([Android Developers][7])
* 发送端需具备基础异常处理：授权失败、采集停止、编码器初始化失败、推流中断。MediaProjection 官方要求在会话停止时正确处理回调与资源释放。([Android Developers][8])

## 6. 非目标

* 不做接收端 UI / 播放器。
* 不做 Chromecast / AirPlay / Meta 官方投屏兼容。
* 不做商业化、账号体系、支付、上架流程。

## 7. 交付物

* Quest 3 发送端 App 原型。
* 单路音视频采集、编码、WebRTC 推流打通。
* 至少一套默认编码配置可稳定输出。
* 预留多编码格式切换接口。

## 8. 实现建议

* **优先实现 Android App 版本**，OpenXR App 作为后续增强形态。Meta Horizon OS 原生支持 Android 应用开发；Quest 同时支持 OpenXR。([Meta for Developers][1])
* 第一优先级：**MediaProjection → MediaCodec(H.264) → WebRTC**。
* 第二优先级：补 **AudioPlaybackCapture**。
* 第三优先级：补多编码格式能力协商。([Meta for Developers][5])
