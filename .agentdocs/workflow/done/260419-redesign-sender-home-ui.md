## 背景
- 当前 sender 首屏是单页发送控制台，语言、扫码、地址、会话 ID、音频、编码、分辨率、帧率、码率与状态全部同时出现，首次使用时信息密度过高。
- 本轮目标不是改协议或推流能力，而是把首页收敛成“开箱即用”的连接入口：用户一眼只需要知道可以扫码连接，也可以手动输入 IP/端口连接。
- 手动连接仍需兼容现有 `ws/wss + path + sessionId` 推流协议，因此首页只暴露高频字段，低频连接细项与语言/规格配置统一下沉到全屏二级设置页。
- 发送端视觉需要和仓库里的蓝绿色、圆润几何、轻玻璃感设计语言对齐，同时保持 Quest 窗口场景下的响应式可读性。

## 关键决策
- 首页手动连接固定只展示 `IP/域名` 与 `端口` 两个字段，允许输入局域网地址或域名，不再直接展示完整 `ws://` / `wss://` 地址。
- `scheme`、`path`、`sessionId`、语言、音频、编码与视频规格全部移到全屏“更多设置”页；首页只保留进入设置的次级操作。
- 不引入新的导航库；页面切换在 `feature/stream-control` 内部通过 Compose 状态与动画完成。
- 底层真实配置仍然只认 `StreamConfig.signalingEndpoint`，UI 层负责在首页 host/port 输入与完整 endpoint 之间做双向映射。

## 验收口径
- 首屏默认可直接理解“扫码连接”与“手动连接”两条路径，不需要阅读大段技术说明。
- 扫码后仍会自动回填配置并立即开始推流。
- 从持久化配置恢复时，首页能正确展示 host/port，设置页能正确展示 `scheme/path/sessionId` 与规格选项。
- 推流前仍可修改设置，推流中保持已有只读/禁用约束。

## 实现结果
- `feature/stream-control` 将旧的单页控制台重构为首页与全屏设置页两段式界面：首页只保留扫码连接、IP/端口手动连接、状态与进入更多设置；设置页承接语言、`scheme/path/sessionId`、系统音频、编码和视频规格。
- 新增 `SignalingEndpointDraft` 作为 UI 层连接草稿，把首页 `IP/域名 + 端口` 输入与底层 `signalingEndpoint` 做双向映射，同时保持扫码回填和持久化恢复链路不变。
- Compose 结构按首页、设置页、共享组件拆分，补充页面切换动画、蓝绿色玻璃感背景与统一的扫码覆盖层视觉，避免旧控制台式信息堆叠。
- 单元测试补齐 endpoint draft 解析、ViewModel 草稿持久化映射、首页/设置页 UI 交互与新测试标签断言。
